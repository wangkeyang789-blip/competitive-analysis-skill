# 竞品分析报告生成器

生成专业的、结构化的竞品分析报告，覆盖竞品筛选、多维度分析和可执行的结论建议。

**本 skill 的核心设计思路：**
- 先对齐分析框架，再展开细节——框架不对，分析白做
- 分两个阶段交付：第一版对齐框架，第二版展开报告
- 帮小白用户补分析方法——怎么选竞品、怎么交叉对比、怎么识别市场空白
- 标注信息来源可信度，不编造数据

---

## 第零步：判断用户是否适合用这个 skill

> 启动前的必要检查。本 skill 不是市场科普工具，它适合有具体分析目标的用户。

**先问用户这一句话：**
> "你想分析哪个产品/行业？大概想通过这次分析解决什么问题？比如是想找产品方向、了解竞争格局，还是为某个决策找依据？"

**判断标准：**

| 用户的回答 | 判断 | 处理方式 |
|-----------|------|---------|
| 能说出具体行业或竞品 + 分析目的（如"分析一下国内笔记类App的格局"） | ✅ 有明确目标，可以推进 | 进入第一步 |
| 有方向但模糊，比如「帮我看看 AI 赛道有什么机会」或「我想了解一下在线教育市场」 | ⚠️ 范围太宽，需要引导 | 追问明确具体赛道/问题边界，再进入第一步 |
| 完全没方向，比如「我不知道该分析什么，你帮我想想」 | ❌ 不适合此时用本 skill | 温和说明：竞品分析需要先有具体问题或方向，建议先想清楚「我想了解哪个市场/什么决策需要依据」，再回来做结构化分析 |

---

## 识别用户当前的阶段

| 用户的情况 | 进入模式 |
|-----------|---------|
| 有明确分析对象（产品/行业），从零生成报告 | **→ 模式 A：分两版生成** |
| 已有分析初稿或零散素材，需要整理成结构化报告 | **→ 模式 B：评估 + 改进** |
| 已有完整报告，需要补充新竞品或新维度 | **→ 模式 C：增量融合更新** |

---

## 模式 A：分两版生成

### 第一步：分析框架对齐

在花时间搜索和撰写完整报告之前，先和用户对齐分析框架。**框架不对，后面的分析再细也是浪费。**

#### A. 明确分析目标

问用户 1-2 个问题，搞清楚这次分析的用途：

- 这次分析主要是为了什么？（找产品方向 / 了解竞争格局 / 为融资或汇报准备材料 / 做功能对标 / 其他）
- 你目前对这个市场的了解程度如何？（完全陌生 / 有一定了解 / 很熟悉）

#### B. 确定竞品范围

帮用户筛选竞品，而不是全盘接受用户随口说的名字：

- **用户提名**：让用户说出他知道的竞品
- **搜索补充**：用 WebSearch 搜索行业报告和市场份额数据，补充用户没提到的竞品
- **筛选原则**：建议选 3-5 个竞品，覆盖以下类型（如果有）：
  - 市场头部（份额最大）
  - 直接竞品（最相似）
  - 差异化选手（模式不同但有参考价值）
  - 新进入者（增长快或模式新）

输出竞品候选列表，并说明每个竞品为什么选/为什么不选，让用户确认。

#### C. 确定分析重点维度

从以下维度中，让用户选出这次最关注的 3-5 个（不是所有维度都要平均用力）：

- 产品定位与目标用户
- 商业模式与收入
- 核心功能与产品设计
- 用户规模与市场数据
- 用户反馈与口碑
- 技术能力与差异化

**帮小白用户做判断：** 如果用户说「你看着定」，根据分析目标主动推荐重点维度：
- **找产品方向** → 重点看用户反馈（槽点=机会）+ 功能差异
- **了解竞争格局** → 重点看市场数据 + 商业模式 + 定位
- **功能对标** → 重点看产品设计 + 技术能力

#### D. 输出【第一版 · 分析框架】

```
# 【行业/产品名】竞品分析框架

## 分析目标
- 本次分析的目的：
- 核心要回答的问题：

## 竞品列表
| 竞品 | 入选理由 | 类型（头部/直接/差异化/新进入者） |
|-----|---------|-------------------------------|
| 竞品A | | |
| 竞品B | | |
| 竞品C | | |

## 重点分析维度
- 🔴 核心维度 1（原因）
- 🟡 核心维度 2（原因）
- ⚪ 补充维度（简要覆盖即可）

## 不做/不深入的方向
- 哪些相关但这次不分析，以及原因

## 数据来源计划
- 预期通过搜索获取的信息类型
- 用户自己能提供的资料
```

输出后询问用户：
> "这个框架符合你的预期吗？竞品选择、分析重点——有没有要调整的？确认后我开始展开完整报告。"

- 用户确认 → 进入第二步
- 用户调整 → 修改框架，重复直到对齐

### 第二步：输出【第二版 · 完整分析报告】

> ⚠️ **只有分析框架得到用户明确确认后，才能进入这一步。**

#### 1. 竞品介绍

- 市场背景：行业现状、用户痛点、市场趋势
- 本次分析涉及的主要竞品
- 竞品分析目的（要点列表）

#### 2. 竞品介绍

每个竞品需要包含：
- 成立时间
- 平台模式（C2C/B2C/B2B 等）
- Slogan
- 产品介绍（核心功能，要点列表）
- 商业模式（收入来源）

#### 3. 产品定位对比

| 对比维度 | 竞品A | 竞品B | 竞品C |
|---------|------|------|------|
| 目标用户群体 | | | |
| 核心价值主张 | | | |
| 市场定位策略 | | | |
| 品牌认知 | | | |

#### 4. 商业策略对比

| 对比维度 | 竞品A | 竞品B | 竞品C |
|---------|------|------|------|
| 收入模式/定价 | | | |
| 增长策略 | | | |
| 合作/分发渠道 | | | |
| 成本结构 | | | |

#### 5. 产品设计对比

| 对比维度 | 竞品A | 竞品B | 竞品C |
|---------|------|------|------|
| 核心功能 | | | |
| 交互/用户体验思路 | | | |
| 关键产品决策 | | | |
| 技术差异化 | | | |

#### 6. 市场数据对比

| 指标 | 竞品A | 竞品B | 竞品C | 数据来源 |
|-----|------|------|------|---------|
| 用户规模 | | | | |

#### 7. 用户反馈对比

| 对比维度 | 竞品A | 竞品B | 竞品C |
|---------|------|------|------|

- 各竞品的关键差异
- 反映出的行业趋势
- 市场集中度/竞争格局判断

- 每个竞品值得学习的地方

##### 9.2 机会点
- 市场空白或未满足的用户需求

- 可差异化的方向
- 建议的后续行动

## 模式 B：评估 + 改进已有素材

用户已经有分析初稿或零散素材时，按以下三层检查：

### 框架层
- [ ] 分析目标明确（知道为什么做、给谁用）
- [ ] 竞品选择有理由（不是随便凑的）
- [ ] 分析维度覆盖了定位/产品/用户/商业中最少 3 个方面
- [ ] 结论和建议有依据，不是主观判断

### 内容层
- [ ] 每个竞品的信息覆盖一致（不是有的写很多有的几乎没写）
- [ ] 对比用到了表格而不是纯段落
- [ ] 数据/论断标注了来源或可信度
- [ ] 用户反馈有具体依据（不只是笼统说"口碑不错"）

### 洞察层
- [ ] 不只停留在"描述竞品在做什么"，有分析"为什么"
- [ ] 机会点具体可执行（而不是"要加强用户体验"这种套话）
- [ ] 指出了竞品的薄弱环节或市场空白

**输出格式：**
1. 总体评价 + 一句话总结
2. 分层列出具体问题（哪个维度缺什么）
3. 问用户：「我可以直接补全缺失的部分，或者你想先自己改——更倾向哪种？」

## 模式 C：增量融合更新

用于用户在已有报告基础上补充新竞品或新维度。

### 操作步骤

1. **让用户提供现有报告**（粘贴或告知文件路径）
2. **明确用户想补充什么**：新竞品 / 新维度 / 更新数据
3. **定位影响范围**：
   - 加新竞品 → 更新所有对比表格 + 综合总结 + 结论
   - 加新维度 → 新增对比表格 + 更新总结
   - 更新数据 → 定位到对应表格和结论
4. **执行融合**：在原有内容基础上增加，不删除原有内容
5. **标注改动**：用 `【本次更新】` 标注所有新增/修改的部分

### 融合后检查
- 新旧竞品的信息覆盖维度是否一致？
- 综合总结是否反映了新增内容？
- 结论建议是否需要根据新信息调整？

---

## 数据来源规则

- **自动模式**：用 WebSearch 为每个章节搜集信息。搜索竞品背景、市场数据、用户评价、产品功能
- **混合模式**：用户提供部分资料，搜索补充缺失部分
- **资料模式**：用户提供全部调研材料，基于已有资料整理，不额外搜索
- **可信度标注**：
  - `[数据来源：...]` — 有明确来源
  - `[估算]` — 基于公开信息的合理推测
  - `[无法获取]` — 搜索后仍无法找到
  - `[用户需补充]` — 依赖用户掌握的私有信息
- **每个竞品**在自动模式下至少需要 2-3 个不同来源的信息
- 如果某个章节搜索不到足够信息，标注 `[数据缺口]` 而非编造

## 输出格式

### Markdown
- 完整报告在对话中展示
- 对比章节优先使用表格
- 功能和详情使用要点列表
- 核心发现用加粗或引用突出

### Word 文档（.docx）
- 保存到 `competitive-analysis/[产品名]-竞品分析.docx`
- 先把 Markdown 报告保存为临时 `.md` 文件
- 用内置脚本转换：`python scripts/markdown_to_docx.py <input.md> <output.docx> --title "报告标题"`
- 如果默认 `python` 不可用，尝试 `/c/Users/USER/miniconda3/python.exe`
- 保留标题层级、表格和列表

## 通用原则（每次执行都必须遵守）

1. **先对齐框架，再展开分析**
   竞品列表和分析维度没有得到用户确认前，不进行大规模搜索和报告撰写。顺序是：分析目标确认 → 竞品筛选 → 维度确认 → 框架对齐 → 完整报告展开。

2. **多源交叉验证，标注可信度**
   每个关键数据/论断尽量有多个来源支撑。无法确认的数据明确标注可信度，不靠猜测填内容。

3. **用对比表格替代纯段落**
   所有涉及「多个竞品 × 多个维度」的内容用对比表格呈现，让差异一目了然。综合总结用矩阵+文字解读，而不是纯段落。

4. **帮小白用户补分析方法**
   用户通常不会主动想到「怎么选竞品」「哪些维度值得看」「怎么判断市场空白」——这些是分析方法的盲区，主动帮用户补上。

5. **洞察 > 描述**
   不止步于描述「竞品做了什么」，要分析「为什么这么做」「这意味着什么」「对我们有什么启示」。描述是信息，洞察才是价值。

6. **融合不覆盖**
   追加分析时，新内容融合进旧报告，不删除原有内容。用 `【本次更新】` 标注改动。

7. **宁可标 [数据缺口] 也不编造**
   信息不足时，明确标出 `[数据缺口]`、`[估算]` 或 `[无法获取]`，不用模糊语言糊弄。

输出完整报告前确认：

- [ ] 每个竞品的覆盖维度一致
- [ ] 对比维度使用了表格代替纯段落
- [ ] 核心发现包含分析解读（不只是罗列事实）
- [ ] 论断有搜索依据或明确标注了可信度
- [ ] 结论和建议具体可执行
- [ ] Markdown 格式正确
- [ ] Word 文档生成成功并已保存
